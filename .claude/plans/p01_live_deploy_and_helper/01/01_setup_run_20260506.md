## Problem Statement

### Problem_1 — Get the 0.1.1 stack actually running on a real workspace

- **User Story**: As the project owner, I want `packages/0.1.1/` deployed end-to-end against my Free Edition workspace so I can run the demos and validate the design empirically (not just on paper).
- **Description**: The framework plan delivered all SQL artifacts + a thin runbook helper. None of it has been executed against a live Databricks workspace. Live execution will surface bugs that no amount of static review catches (and did — see `CAST(source_key AS BIGINT)` in DECISIONS #16).
- **Impact**: Without a live deploy, the demos are claims rather than evidence; later iterations (0.1.2 non-PD teams, 0.1.4 SCHEDULE clauses, etc.) build on an unverified base.
- **Acceptance Criteria**:
  - [x] Connection to live workspace established + verified (`SHOW CATALOGS`).
  - [x] Catalog `medallion_demo` created idempotently.
  - [x] Pre-bronze schemas + raw tables + seed populated (~23K positions, ~18K transactions).
  - [x] Bronze: 14 t + 14 v + 14 mv + crosswalk + lineage_audit + 15 refresh procs.
  - [x] Silver: 18+6 t + 18+6 v + 18+6 mv + 19 refresh procs + documentation.
  - [ ] Gold: 53 t + 53 v + 53 mv + 60 refresh procs (mv creation may be deferred — open decision below).
  - [ ] Orchestrator + `CALL bronze_silver_gold_refresh()` runs clean.
  - [ ] `05_validate/01..06.sql` all PASS.
- **Notes**:
  - Workspace: AWS, `dbc-40c058d4-649b.cloud.databricks.com`, region us-east-2.
  - Token used in this session was pasted in chat. User noted (2026-05-07): "free demo account for learning"; rotation is a close-out housekeeping item, not a security concern at this scope.

### Problem_2 — Stop hand-rolling `curl + jq` for every Databricks call

- **User Story**: As the future-self running this stack, I want a small helper library so submitting a SQL file to the warehouse is one command instead of ~30 lines of bash.
- **Description**: Every Databricks API call I made by hand was a fresh `curl + jq` block. Each new call repeated the same auth header / payload-build / poll-loop / result-parse logic. The bash heredocs broke under multi-line strings; jq broke on Databricks responses with unescaped control chars.
- **Impact**: Without a helper, every iteration cost 5–10 min of bash debugging. With one, it's `python3 scripts/dbx_run.py file.sql`.
- **Acceptance Criteria**:
  - [x] Helper handles compound-block wrapping (`BEGIN..END;`) and `DECLARE OR REPLACE` → `DECLARE` rewrite for the SQL Statement Execution API.
  - [x] Backend uses `subprocess + curl` (system trust store; works through corp SSL inspection where Python `urllib` fails).
  - [x] Polls until terminal state without arbitrary timeouts.
  - [x] Retries on transient curl errors (codes 6, 7, 28, 35, 52, 56) with backoff.
  - [x] CLI surface: file run, inline SQL, `--no-wrap`, `--continue`, `--quiet`.
  - [x] `runbook.sh` refactored to delegate to `dbx_run.py`; runbook order is one bash array.
  - [x] `scripts/README.md` documents the architecture.

### Out of Scope

- A full Asset Bundle deploy job (deferred to 0.1.7 milestone; the `databricks.yml` stub stays a placeholder).
- A first-class Python package (current shape: two scripts + a README is sufficient until we need import-from-elsewhere).
- Per-MV parallelism (Free Edition's 1-pipeline cap means it would not help).

## Risk & Mitigation

### Risk & Mitigation Strategy

- **Risk: token leak.** The Databricks PAT was pasted into chat. Per user (2026-05-07), this is a Free Edition demo account; risk is bounded. Mitigation deferred to plan close-out: rotate via Databricks UI, store new token in macOS Keychain (`security add-generic-password`), add to `SECRETS` in `~/.claude/load-secrets.sh`.
- **Risk: client-side timeout during long MV runs.** Mitigation already in: dbx_run.py retries transient curl failures; server-side execution continues even if client disconnects; re-run with `CREATE OR REPLACE` is idempotent.
- **Risk: more silent SQL bugs in silver/gold cancels / bridge views.** Mitigation: validate harness (`05_validate/`) catches FK-orphan + SCD2-overlap classes on the next pass; manual review when each MV fails surfaces issues early.
- **Risk: warehouse auto-stop mid-deploy.** Default 10-min idle stop; long sleeps inside the deploy could trigger stop + cold restart. Mitigation: dbx_run.py polls every 4s, keeping warehouse warm; also acceptable to stop/start.

### Rollback Plan

- `pnpm run nuke` (NUKE_CONFIRM=YES) drops the catalog cleanly. From there, re-deploy from scratch.
- All DDL is idempotent (`CREATE OR REPLACE` / `IF NOT EXISTS`), so partial state is recoverable.
- Helper code is git-tracked; revert to commit `f46fdb3` to restore the pre-live-deploy state.
